Astro Island Hydration & EaseMotion Class Timing

EaseMotion's classes are plain CSS, so they render the instant the stylesheet loads. But if a hover or click needs a component to add or remove that class, Astro's island hydration model means the interaction isn't live until the island actually hydrates — client:load, client:idle, and client:visible all hydrate at different times. This guide shows the gap and how to design around it.

Astro's hydration directives, at a glance

Each directive controls when a component's JavaScript runs — not when its CSS applies. EaseMotion classes present in the markup are already styled before any of these fire.

Directive Hydrates when… Good fit for
client:load Immediately, as soon as the page loads. Above-the-fold interactions the visitor needs right away.
client:idle Once the main thread is free (via requestIdleCallback). Useful, non-urgent interactivity that shouldn't compete with initial paint.
client:visible When the component scrolls into the viewport. Below-the-fold islands — the usual default for performance.
client:media When a given CSS media query first matches. Interactivity that only matters at certain breakpoints (e.g. a mobile-only menu).
client:only Client-side only — skips server rendering entirely. Components that can't render meaningfully without a browser API.

Problematic vs recommended timing

The fix is usually to stop asking JavaScript to do something CSS can already do on its own.

✗ Hover effect gated on hydration
---
// Card.astro
---
<div client:visible>
  <CardReact client:visible />
</div>

// CardReact.jsx
<div
  className={hovered ? "ease-hover-grow" : ""}
  onMouseEnter={() => setHovered(true)}
  onMouseLeave={() => setHovered(false)}
>

The hover class only gets applied once React state updates — which can't happen until the component hydrates. Until then, hovering silently does nothing, which reads as a bug rather than a loading state.

✓ Hover handled entirely in CSS
---
// Card.astro
---
<div class="ease-card ease-hover-grow">
  <CardReact client:visible />
</div>

/* No hover state in JS at all —
   :hover is native, so it works
   before the island ever hydrates. */

The hover animation lives entirely in the ease-* classes and a native :hover rule. client:visible is still used for whatever actually needs React (form state, data fetching), but the motion itself never waits on it.

Copy-ready integration snippet

A minimal pattern: CSS-driven entrance and hover on the wrapper, hydration reserved for whatever genuinely needs JavaScript inside it.

ProductCard.astro 
---
import AddToCartButton from "./AddToCartButton.jsx";
const { product } = Astro.props;
---
<article class="ease-card ease-fade-in ease-hover-grow">
  <img src={product.image} alt={product.name} />
  <h3>{product.name}</h3>

  <!-- Only this part needs JS, so only this part hydrates -->
  <AddToCartButton client:visible productId={product.id} />
</article>

Choosing a strategy by interaction criticality

  • Purely decorative — entrance animations, hover states, focus rings: keep these in CSS with ease-* classes and skip hydration for them entirely.
  • Above-the-fold and essential — a hero CTA that must respond instantly: client:load.
  • Useful but not urgent — a "save for later" button below a long article: client:idle.
  • Below the fold — most cards, accordions, and secondary widgets: client:visible, so hydration cost is paid only if the visitor scrolls that far.
  • Breakpoint-specific — a mobile nav toggle that doesn't exist on desktop: client:media="(max-width: 768px)".
  • Browser-API-dependent — anything that can't render meaningfully server-side: client:only, used sparingly since it delays first paint of that component.
