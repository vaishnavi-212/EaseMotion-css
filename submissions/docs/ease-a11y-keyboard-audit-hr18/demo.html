Keyboard Accessibility Audit

An audit of the interactive patterns EaseMotion's documentation shows — buttons, modals, dropdowns, accordions, tooltips — against keyboard and screen-reader expectations, paired with a working, accessible reference implementation of each.

This submission doesn't edit core/ or components/ directly — per the repo's current contribution policy, those are temporarily restricted to the maintainer while the framework stabilizes. Instead, each pattern below is a corrected, drop-in reference the maintainer (or any contributor) can use to patch the real components.

Button

role: button (native)
  • Common issue Interactive elements built from <div> or <span> aren't focusable or triggerable by keyboard at all.
  • Common issue Custom focus styles removed with outline: none and never replaced.
  • Fix Use a native <button> — it's focusable, triggerable by Enter/Space, and announced correctly by default.
  • Fix Style focus with :focus-visible instead of removing it, so it only shows for keyboard users.

Tab to this button — the focus ring is visible and never suppressed.

Modal / Dialog

role="dialog" aria-modal="true"
  • Common issue Focus stays on the page behind the modal, so keyboard users can tab into content they can't see.
  • Common issue Escape doesn't close it, and there's no way back to the trigger without a mouse.
  • Fix Move focus into the dialog on open, trap Tab within it, and return focus to the trigger on close.
  • Fix Listen for Escape globally while open, and label the dialog with aria-labelledby.

Open it, then try Tab (focus stays inside) and Escape (closes and returns focus here).

Dropdown menu

role="menu" / role="menuitem"
  • Common issue Menu only opens on hover/click, so keyboard users can't reach the items at all.
  • Common issue No aria-expanded on the trigger, so screen readers don't announce open/closed state.
  • Fix ↓/↑ move between items, Enter selects, Escape closes and returns focus to the trigger.
  • Fix Trigger gets aria-haspopup="true" and aria-expanded kept in sync with the menu's state.

Open with Enter/Space, navigate with arrow keys, close with Escape.

Accordion

aria-expanded / aria-controls
  • Common issue Header built as a clickable <div>, so it's invisible to keyboard and screen reader users.
  • Common issue No relationship exposed between the header and the panel it controls.
  • Fix Each header is a real <button> with aria-expanded and aria-controls pointing at its panel.
  • Fix The panel carries a matching id, so the relationship is programmatic, not just visual.

Tab to a header, press Enter or Space to toggle it.

Tooltip

aria-describedby
  • Common issue Tooltip only shows on :hover, so it never appears for keyboard-only users tabbing through the page.
  • Common issue No text relationship to the trigger, so screen readers never announce it at all.
  • Fix Show on :focus-visible as well as :hover, and let Escape dismiss it without moving focus away.
  • Fix Connect trigger and bubble with aria-describedby so the tooltip text is announced with the control.

Tab to the button — the tooltip appears; press Escape to dismiss it early.

What every pattern above does

✓ Focus visible — every interactive element uses :focus-visible, never suppressed.
✓ Escape closes dialogs — modal, menu, and tooltip all dismiss on Escape.
✓ Logical tab order — no tabindex greater than 0 anywhere; DOM order matches visual order.
✓ ARIA roles/labels — dialog, menu, menuitem, and tooltip roles applied with matching labelling attributes.
