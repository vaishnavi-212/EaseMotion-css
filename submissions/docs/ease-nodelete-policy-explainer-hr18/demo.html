Delete-Detection Explainer

Most closed pull requests here aren't closed for bad code — they're closed for touching the wrong files. This page walks through what a valid submission diff looks like next to an invalid one, so you can check your own PR before opening it.

Invalid vs valid, side by side

Both examples below are illustrative — the file paths are representative, not copied from a real PR.

✗ INVALID core/animations.css
@keyframes ease-fade-in {
- from { opacity: 0; }
+ from { opacity: 0; transform: scale(0.9); }
to { opacity: 1; }
}

Why this gets closed: core/animations.css is an existing framework file. Editing it — even by one line — is a direct edit to shared infrastructure, which only the maintainer can review and merge safely.

✓ VALID submissions/examples/your-feature-hr18/style.css
+.your-feature-hr18 {
+ animation: ease-fade-in-hr18 400ms ease both;
+}
+
+@keyframes ease-fade-in-hr18 { … }

Why this is valid: every line is a new addition inside a brand-new folder under submissions/. Nothing existing is touched, so it can't conflict with anyone else's work or the core framework.

The rules a diff gets checked against

These are the checks most first-time contributors miss.

01

New folder only

Your submission must live in a brand-new folder you create — never inside an existing one, and never as loose files dropped into submissions/ itself.

02

Unique folder name

Suffix your folder with your own initials or handle (e.g. -hr18) so it can't collide with someone else's submission for the same issue.

03

core/ and components/ are off-limits

Any diff touching either directory is treated as an invalid, unreviewed edit to shared infrastructure and is closed on sight.

04

3-file minimum

A Standard-track submission needs at least demo.html, style.css, and README.md — a single file isn't a complete, reviewable submission.

05

Line-count sanity check (≥250)

PRs that add far fewer lines than that across a full submission are often a sign something's missing (styles, docs, or a working demo) — it's a smell to double-check, not a hard line to pad out.

06

One feature per PR

Don't bundle an unrelated fix or a second feature into the same branch — it makes the diff harder to review and easier to reject outright.

Valid folder structure, by track

Pick the tab for the track you're contributing to.

submissions/examples/your-feature-hr18/ ├── demo.html ├── style.css └── README.md
submissions/docs/your-guide-hr18/ ├── demo.html ├── style.css └── README.md
submissions/react/your-component-hr18/ ├── App.jsx ├── App.scss ├── package.json └── README.md
submissions/scss/your-mixin-hr18/ ├── _index.scss ├── demo.html └── README.md

Common reasons a PR gets auto-closed

  • 01 Diff includes changes inside core/ or components/.
  • 02 Files placed directly in submissions/ instead of a new subfolder.
  • 03 Folder name reused from an existing submission (naming collision).
  • 04 Missing one of the required files for the track (e.g. no README.md).
  • 05 Multiple unrelated features bundled into a single PR.
  • 06 Commit history left unsquashed when the contributing guide asks for one commit.

Before you open your PR

  • All new files live inside one new folder under submissions/<track>/.
  • Folder name has a unique suffix (your initials or handle).
  • No files under core/ or components/ were touched.
  • Required files for the track are all present (see the tabs above).
  • Only one feature or fix in this PR.
  • PR description links the issue it resolves.
