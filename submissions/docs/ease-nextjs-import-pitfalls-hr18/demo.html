Next.js App Router Import Placement Pitfalls

EaseMotion is a single global stylesheet import, but where you put that import in a Next.js App Router project changes what a visitor actually sees. Import it in the wrong file and you can get a flash of unstyled content (FOUC) every time someone navigates. This guide shows why, and where the import actually belongs.

Wrong vs correct import placement

The App Router re-renders page.js on every navigation to that route, but keeps a shared layout.js mounted across all of them — which is exactly why import location matters here in a way it didn't in the old Pages Router.

✗ app/page.js
// app/page.js
import "easemotion-css/min";

export default function Page() {
  return (
    <main className="ease-fade-in">
      <h1>Home</h1>
    </main>
  );
}

Works on the first load, but every other route (app/about/page.js, etc.) needs its own copy of this import — miss one and that page silently renders unstyled. Even where it is imported, Next.js can reload the chunk on navigation, producing a brief unstyled flash.

✓ app/layout.js
// app/layout.js
import "easemotion-css/min";

export default function RootLayout({ children }) {
  return (
    <html lang="en">
      <body>{children}</body>
    </html>
  );
}

Imported exactly once, in the file that wraps every route. The stylesheet is applied before any page renders and stays applied across client-side navigations, since the root layout never unmounts.

Why the flash happens — and how to prevent it

Cause: per-route imports

A CSS import inside page.js is scoped to that route's bundle. On navigation, React has to mount the new page (and its styles) before they can apply — there's a real, if brief, unstyled window.

Cause: duplicate imports

Importing the same stylesheet in multiple page files doesn't error, but it does nothing for consistency — each route still only gets styled once its own bundle loads.

Prevention: import once, at the root

Put the import in app/layout.js (or app/globals.css's own import chain). It loads with the shell that wraps every page, so there's nothing left to flash.

Prevention: don't nest it in Client Components

Keep the import in a Server Component layout. Pushing it into a deeply nested "use client" component ties the stylesheet to that component's hydration timing instead of the initial HTML response.

Server Component compatibility

EaseMotion's classes are plain CSS, so importing the stylesheet needs no "use client" directive and works fine inside a Server Component like the default app/layout.js. You only need "use client" for a component if it also uses React state, effects, or browser-only APIs — not just because it happens to have EaseMotion classes on its markup.

Copy-ready imports

Pick the file you're editing.

// app/layout.js
import "easemotion-css/min";

export default function RootLayout({ children }) {
  return (
    <html lang="en">
      <body>{children}</body>
    </html>
  );
}
// app/page.js
// No stylesheet import here — it already loads
// once, globally, from app/layout.js.

export default function Page() {
  return (
    <main className="ease-fade-in">
      <h1>Home</h1>
    </main>
  );
}

Pitfall checklist

  • Import order — if you also have a custom globals.css, import EaseMotion before it so your own overrides win the cascade.
  • Global scope — the import should appear exactly once, in app/layout.js, not repeated per route.
  • Duplicate imports — search your app/ tree for other easemotion-css imports before adding a new one; a second import doesn't break anything but is a sign the first one is in the wrong place.
  • No unnecessary "use client" — importing the stylesheet doesn't require marking the file a Client Component.
